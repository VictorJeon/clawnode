# 오픈클로 스킬 만들기: 좋은 스킬 vs 나쁜 스킬

ClawHub에는 현재 3,286개의 스킬이 등록되어 있고, 총 다운로드 수는 150만 회를 넘었습니다.

npm이나 Chrome 웹스토어 같은 생태계가 떠오릅니다. 그런데 근본적으로 다른 점이 있습니다.

**스킬은 코드가 아닙니다.** SKILL.md라는 텍스트 파일 하나입니다.

폰에 앱을 설치하면 새 프로그램이 실행됩니다. AI에게 스킬을 설치하면 새 교육 매뉴얼이 전달됩니다. AI의 능력 자체가 바뀌는 게 아니라, AI가 아는 것이 바뀝니다.

## 스킬과 플러그인: 가장 많이 혼동하는 부분

커뮤니티에서 자주 올라오는 질문:

> "index.js를 만들었는데 왜 도구로 등록이 안 돼요?"

스킬과 플러그인은 다릅니다.

| | 스킬 | 플러그인 |
|---|---|---|
| 형태 | SKILL.md 텍스트 파일 | JS/TS 실행 코드 |
| 제작 시간 | 5~30분 | 개발 필요 |
| 바꾸는 것 | AI가 아는 것, 행동 방식 | 오픈클로가 할 수 있는 것 |
| 비유 | 비서에게 새 매뉴얼 주기 | 비서 책상에 새 전화기 놓기 |

스킬은 비서에게 "이런 상황이 오면 이렇게 해"라고 교육하는 것입니다. 플러그인은 비서에게 새 도구(팩스, 스캐너)를 갖다주는 것입니다.

이 구분을 모르면 스킬로 해결할 수 있는 걸 플러그인으로 만들거나, 그 반대로 시간을 낭비합니다.

## SKILL.md 안에는 뭐가 있나

잘 만든 스킬의 SKILL.md 구조입니다.

```markdown
# 스킬 이름

## 설명
이 스킬은 무엇을 하는가 (1~2문장)

## 트리거
이 스킬이 활성화되는 상황
- "날씨 알려줘"라고 할 때
- 위치 기반 질문이 들어올 때

## 사용법
1. 이렇게 한다
2. 저렇게 한다

## 제약
- 이건 하지 마라

## 의존성
- 필요한 CLI: weather-cli
- 필요한 API 키: 없음
```

각 섹션을 비개발자에게 설명하면:
- **설명**: 어떤 업무용 매뉴얼인지
- **트리거**: 언제 이 매뉴얼을 꺼내봐야 하는지
- **사용법**: 업무 처리 절차
- **제약**: 하면 안 되는 것
- **의존성**: 이 업무에 필요한 도구

전체 카테고리에서도 AI/ML이 1,588개로 최다입니다. "자기 자신을 바꾸는 도구"가 가장 인기 있는 카테고리라는 아이러니입니다.

## 스킬 우선순위: 같은 이름이 여러 곳에 있으면

같은 이름의 스킬이 여러 위치에 있을 때 우선순위는 이렇습니다.

1. `workspace/skills` (최우선) — 프로젝트 전용
2. `~/.openclaw/skills` (중간) — 내 계정 전체
3. 번들 스킬 (최하위) — 오픈클로 기본 내장

상위가 하위를 덮어씁니다. 회사 매뉴얼이 본사/지사/팀 세 곳에 있을 때, 팀 매뉴얼이 최우선인 것과 같습니다.

중요한 함정이 있습니다. `skills.watcher.enabled`가 false면 새 스킬을 설치해도 세션에 반영되지 않습니다. "스킬이 작동 안 해요"의 90%는 이 watcher 설정 문제입니다.

## "읽었는데 안 따라요"

커뮤니티에서 반복적으로 나오는 불만입니다.

> "Opus 4.6이든 GPT 5.2든, 스킬을 읽어도 스킬 지시를 안 따른다."

원인은 세 가지입니다.

**모델의 자체 판단 우선**: 경험 많은 모델일수록 자기 판단을 신뢰합니다. 매뉴얼을 무시하는 경력직처럼, 스킬 지시보다 모델 자체 판단을 따릅니다.

**스킬 지시의 모호함**: "친절하게 답하되 간결하게"는 상황마다 해석이 다릅니다. 구체적이지 않은 지시는 모델이 자기 방식으로 해석합니다.

**컴팩션 후 맥락 손실**: 긴 대화 후 컴팩션이 발생하면 스킬 관련 맥락이 요약에서 빠질 수 있습니다. 컴팩션과 메모리에 대한 내용은 [AI의 기억상실: 컴팩션은 왜 필요하고 뭘 잃는가](/blog/ai-memory-compaction)에서 다룹니다.

AGENTS.md에 "반드시 스킬을 읽어라"고 명시해도 100% 준수는 불가능합니다. 스킬은 제안이지 강제가 아닙니다. 이건 AI 에이전트의 근본적 한계이자, 단순한 프로그램이 아닌 이유이기도 합니다.

## 잘 만든 스킬 vs 위험한 스킬

커뮤니티가 정리한 품질 기준입니다.

**좋은 스킬의 특징:**
- SKILL.md에 목적, 입력, 출력, 의존성이 명확
- 5분 안에 첫 결과 확인 가능
- 업데이트 이력과 이슈 해결 흔적이 있음
- 필요한 권한만 요청

**위험 신호:**
- 설치 수 100회 미만 + 소스 비공개
- "이 외부 프로그램을 먼저 설치하세요" 지시 (악성 스킬의 전형적 수법)
- 파일 시스템 전체 접근 같은 과도한 권한 요청
- SKILL.md가 모호하거나 다른 스킬을 그대로 복사한 것

5분 안에 결과가 나오지 않으면 나쁜 스킬입니다. AI 스킬에서는 문서의 명확성이 코드 품질보다 중요합니다.

2026년 1월 ClawHavoc 사건에서 335개의 악성 스킬이 발견된 이후, 스킬 설치 전에 SKILL.md를 직접 읽어보는 습관이 필수가 되었습니다. 자세한 내용은 [AI 에이전트 보안](/blog/ai-agent-security)에서 확인할 수 있습니다.

## 스킬이 조용히 사라지는 문제

GitHub Issue #44051, 2026년 3월 업데이트 이후 보고된 내용입니다.

> "symlink 기반으로 설치된 스킬이 'path resolves outside its configured root'로 조용히 스킵됩니다. 에러 로그를 보지 않으면 스킬이 사라진 걸 모릅니다."

보안 강화 업데이트 때문에 기존 스킬이 작동하지 않을 수 있습니다. "잘 되던 게 갑자기 안 됨" 시나리오입니다. 업데이트 후에는 반드시 `/status`로 로드된 스킬 목록을 확인해야 합니다.

## 나만의 스킬 만들기

스킬 제작은 생각보다 간단합니다. SKILL.md 파일 하나면 됩니다.

1. `~/.openclaw/skills/내스킬이름/SKILL.md` 파일 생성
2. 위의 SKILL.md 구조대로 작성
3. `skills.watcher.enabled: true` 설정 확인
4. 오픈클로 재시작 또는 스킬 리로드

ClawHub에 공개하려면 GitHub 리포지토리 생성 후 `clawhub publish` 명령으로 등록할 수 있습니다. 잘 만든 스킬이라면 커뮤니티에서 자연스럽게 퍼집니다.
