"에이전트가 파일을 만들고 배포했다고 했는데, 실제로는 아무것도 없었습니다."

커뮤니티에서 자주 올라오는 보고입니다. 단순한 버그가 아닙니다. AI 에이전트가 가진 구조적 한계입니다.

OpenClaw를 제대로 쓰려면 "잘하는 것"보다 "절대 혼자 믿으면 안 되는 것"을 먼저 알아야 합니다.

## 한계 1: 도구 환각

가장 빈번한 유형입니다.

에이전트가 `read`, `write`, `exec`를 호출했다고 설명했는데, 실제 로그에 해당 도구 호출 기록이 없습니다. 모델이 도구 실행 자체를 상상한 것입니다.

왜 이런 일이 생길까요. 모델은 "그럴듯한 완료 보고"를 생성하도록 훈련되어 있습니다. 실패 상황에서 "실패했습니다"보다 "완료했습니다"를 답할 유인이 있습니다. 도구 호출이 실제로 이루어졌는지와 무관하게 완료된 것처럼 보고할 수 있습니다.

**대응**: `openclaw logs --follow`로 도구 호출이 실제 발생했는지 확인합니다. 결과가 중요한 작업이라면 파일 존재 여부를 추가로 검증합니다.

**원칙**: 보고보다 로그를 믿으세요.

## 한계 2: 자살 패러독스

GitHub Issue #44000.

에이전트가 설정 변경 후 게이트웨이를 재시작하기 위해 `pkill -f openclaw-gateway`를 실행했습니다. 에이전트 자신이 그 프로세스 안에서 돌고 있었기 때문에, 같이 죽었습니다. 이후 재시작 명령은 영원히 실행되지 않았습니다.

에이전트가 프로세스 경계를 이해하지 못하면 생기는 문제입니다. 자기 발밑을 자르는 것입니다.

이 이슈가 깨주는 환상이 있습니다. 에이전트가 인프라 운영까지 알아서 할 것이라는 생각입니다. 운영 작업은 안전한 전용 도구로 제한해야 합니다. [멀티에이전트 운영의 현실](/blog/multi-agent-reality)에서 이 문제를 더 자세히 다뤘습니다.

## 한계 3: 도구가 갑자기 사라지는 현상

GitHub Issue #34810.

```
4시쯤 에이전트가 갑자기 exec/read/write 도구를 잃어버렸습니다.
파일 만들어달라고 하면 "터미널에서 직접 실행하세요"라고 합니다.
```

실전에서 자주 겪는 상황입니다. 가능한 원인은 세 가지입니다.

1. 세션 전환 후 도구 권한 재주입 누락
2. 게이트웨이 상태 불안정
3. 모델의 잘못된 자기 인식 ("나는 도구가 없다")

세 번째 원인이 가장 당혹스럽습니다. 도구를 호출할 수 있는데 없다고 믿는 경우입니다.

**대응**: 새 세션에서 재시도 → `/status`로 도구 목록 확인 → 게이트웨이 상태 점검 순서로 진행합니다.

## 한계 4: 문서와 코드가 다르다

GitHub Issues #43562, #27038.

- 문서: "SOUL.md로 정체성을 정의하세요"
  코드: 시스템 프롬프트 첫 줄에 "personal assistant" 하드코딩

- 문서: "서브에이전트는 SOUL.md 미주입"
  코드: allowlist에 SOUL.md/USER.md 포함

공식 문서를 100% 신뢰하고 시스템을 설계하면, 실제 동작과 어긋나는 경우가 있습니다. 고급 사용자들이 문서와 소스코드와 이슈를 함께 읽는 이유입니다.

SOUL.md를 실제로 어떻게 작동시키는지는 [SOUL.md 가이드](/blog/soul-md-guide)를 참고하세요.

## 한계 5: 자동화의 착시

GitHub Issues #42097, #44052.

크론 잡이 running 상태에서 멈추고, 실행 기록이 비어있고, 대기 중인 잡들이 전부 막히는 상황입니다.

겉으로는 "자동화가 돌아가는 것처럼" 보이지만 실제론 멈춰 있습니다. 자동화는 설정보다 모니터링이 중요합니다. [AI 크론 자동화](/blog/ai-cron-automation)에서 크론이 안 도는 이유와 해결법을 정리했습니다.

## "AGENTS.md에 적어놨는데도 안 지켜요"

커뮤니티에서 자주 나오는 불만입니다.

원인은 세 가지입니다.

- 모델은 규칙을 "법"이 아니라 "조언"으로 해석할 수 있습니다.
- 긴 세션에서 컴팩션으로 규칙 맥락이 손실됩니다. [메모리 컴팩션 문제](/blog/ai-memory-compaction)에서 다뤘습니다.
- 충돌하는 규칙이 있으면 우선순위 판단을 실패합니다.

**해결 방향**: 자연어 규칙은 단순하고 구체하게 작성합니다. 파괴적 작업은 시스템 레벨 게이트로 강제합니다. AGENTS.md는 짧게 유지하고 핵심 규칙만 담습니다.

## 현실적인 운영 원칙 5개

이 모든 한계를 알고 나서도 AI 에이전트는 유용합니다. 다만 신뢰하는 방식이 달라져야 합니다.

1. **AI의 완료 보고는 가설입니다** — 로그, 파일, 상태로 검증하세요.
2. **파괴적 작업은 분리합니다** — 삭제, 전송, 재시작은 승인 단계를 강제하세요.
3. **자동화는 감시와 함께 합니다** — 크론 모니터링 없으면 자동화는 환상입니다.
4. **문서보다 소스가 진실입니다** — 모순이 생기면 코드와 이슈를 확인하세요.
5. **중요 규칙은 파일과 시스템으로 이중화합니다** — 대화에만 두지 마세요.

## 그래서 어떤 팀원인가

OpenClaw는 "무엇이든 알아서 해주는 마법 비서"가 아닙니다.

빠르고 유능하지만 자주 착각하는 신입 팀원에 가깝습니다.

좋은 팀장은 신입을 신뢰하되, 검증 가능한 프로세스를 만듭니다. OpenClaw 운영도 마찬가지입니다. 처음부터 기대치를 현실에 맞게 잡으면, 오히려 훨씬 많은 일을 맡길 수 있습니다.
