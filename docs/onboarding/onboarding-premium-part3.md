### [1:10~1:12] 왜 기획이 먼저인지 설명

- 비개발자 비유:
  - 요구사항 = 여행 체크리스트
  - 기능 목록 = 여행 일정표
  - 코드 = 실제 이동 수단

### [1:12~1:17] 사용자 시나리오 작성
> "개발자 말이 아니라 사용자 말로 적어볼게요."

예시 문장:
- "나는 할 일을 입력해서 추가할 수 있어요."
- "나는 할 일을 완료로 체크할 수 있어요."
- "나는 필요 없는 할 일을 지울 수 있어요."

### [1:17~1:22] 기능 목록 확정
> "지금부터는 만들 기능을 딱 6개로 고정할게요."

```text
1. 할 일 추가 (텍스트 입력 → 추가 버튼)
2. 완료 체크 (체크박스)
3. 삭제 (X 버튼)
4. 필터 (전체/진행중/완료)
5. 다크모드 (토글)
6. 로컬 저장 (새로고침해도 유지)
```

> "이 기능 목록을 AI에게 주면 AI가 초안을 만들어줘요. 이게 바이브코딩이에요."

### Step 1) 프로젝트 생성 (1:25~1:33)

터미널 실행:

```bash
npx create-next-app@latest my-todo --typescript --tailwind --app --src-dir --eslint
cd my-todo
code .   # Cursor로 열기
```

> "명령어는 주문서예요. 외우는 게 아니라 필요할 때 꺼내 쓰면 돼요."

옵션 한 줄 설명:
- `--typescript`: 오타를 빨리 잡아주는 안전벨트
- `--tailwind`: 빠른 스타일링 도구 포함
- `--app`: 최신 Next.js 구조 사용
- `--src-dir`: 코드 폴더 정리 (`src`)
- `--eslint`: 코드 맞춤법 검사기

📸 **[스크린샷: create-next-app 완료 로그]**
📸 **[스크린샷: 프로젝트 폴더가 Cursor에 열린 화면]**

막힘 대응:
- `npx` 에러 시: Node.js 버전 확인 (`node -v`, 18 이상 권장)
- 설치가 멈추면: 터미널 재실행 후 같은 명령 재시도
- Cursor에서 폴더 안 보이면: `File > Open Folder > my-todo`

### Step 2) Cursor AI로 첫 UI 생성 (1:33~1:45)

> "AI는 신입 개발자라고 생각하면 돼요. 지시가 구체적일수록 잘해요."

- `Cmd + I`: 현재 파일 인라인 수정
- `Cmd + L`: 채팅으로 큰 단위 작업 요청

첫 번째 프롬프트(복붙):

```text
이 Next.js 프로젝트에 할 일 관리 앱을 만들어줘.

기능:
- 할 일 추가 (텍스트 입력 + 추가 버튼)
- 완료 체크 (체크박스)
- 삭제 (X 버튼)
- 필터 (전체/진행중/완료)
- 다크모드 토글
- localStorage에 저장

디자인:
- 깔끔하고 모던한 UI
- 모바일 반응형
- Tailwind CSS 사용
```

> "생성된 코드가 100점이 아니어도 괜찮아요. 일단 실행되는 1차 버전이 중요해요."

- AI 제안 확인 후 **Accept** 클릭
- 실행 확인:

```bash
npm run dev
```

- 브라우저에서 `http://localhost:3000` 확인

📸 **[스크린샷: Cursor AI 응답/Accept 버튼]**
📸 **[스크린샷: localhost:3000에서 Todo UI 첫 실행 화면]**

막힘 대응:
- 앱이 안 뜨면: 터미널 에러 첫 줄부터 읽고 복붙
- 포트 충돌 시: `npm run dev -- -p 3001`
- AI가 파일을 잘못 건드리면: Git으로 되돌리기
  ```bash
  git init
  git add .
  git commit -m "init todo base"
  # 문제 발생 시
  git restore .
  ```

## 고객 예상 질문 & 답변 (최소 10개)

### Q1. 코드를 모르는데 이렇게 만들어도 되나요?
> 네, 괜찮아요. 처음엔 “코드 작성자”보다 “요구사항 전달자”가 되는 게 더 중요해요.

### Q2. AI가 매번 다른 코드를 주는데 정상인가요?
> 네, 정상이에요. 그래서 요구사항을 구체적으로 써야 해요. 색상, 크기, 유지해야 할 기능까지 적으면 흔들림이 줄어요.

### Q3. 에러가 나면 제가 잘못한 건가요?
> 아니에요. 개발은 에러를 찾고 고치는 과정이에요. 에러 메시지를 그대로 주면 AI가 잘 도와줘요.

### Q4. 한 번에 기능을 많이 시키면 더 빠르지 않나요?
> 오히려 느려져요. 작은 단위로 지시해야 어디서 깨졌는지 바로 찾아요.

### Q5. localStorage는 서버인가요?
> 아니에요. 브라우저 안의 작은 메모장이라고 보면 돼요. 내 컴퓨터에만 저장돼요.

### Q6. Next.js랑 React 차이가 뭐예요?
> React는 화면을 만드는 도구이고, Next.js는 그 도구를 더 편하게 쓰게 해주는 완성 키트예요.

### Q7. Tailwind를 꼭 써야 하나요?
> 꼭은 아니지만, 초반에는 빨라요. 미리 준비된 스타일 블록이라 빠르게 결과를 확인할 수 있어요.

### Q8. AI가 만든 코드를 믿어도 되나요?
> 100% 믿지 말고 “실행으로 검증”하면 돼요. 동작 확인 + 에러 확인 + 다시 수정이 기본 루프예요.

### Q9. 어떤 프롬프트가 좋은 프롬프트인가요?
> 기능, 디자인, 제약조건(유지할 것), 완료 기준이 들어간 프롬프트가 좋아요.

### Q10. 앱이 너무 못생겼어요. 어떻게 개선하죠?
> “못생김” 대신 구체적으로 말하면 돼요. 예: 카드 그림자 추가, 둥근 정도, 여백, 글자 크기, 색상 코드.

### Q11. Git은 언제 써요?
> 정상 동작 지점마다 저장(커밋)할 때 써요. 문제 생기면 그 지점으로 되돌아갈 수 있어요.

### Q12. 오늘 만든 걸 진짜 서비스로 써도 되나요?
> 가능해요. 다만 로그인/DB/보안이 필요하면 다음 단계(Part 4 이후)에서 확장하면 돼요.
