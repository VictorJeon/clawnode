"AI가 어제 대화를 왜 기억 못 해요?"

AI 에이전트를 처음 써보는 사람들이 가장 먼저 부딪히는 문제입니다. 그런데 흥미로운 건, 이 문제를 해결하는 방식이 사람마다 완전히 다르다는 겁니다.

OpenClaw 커뮤니티에서 관찰된 접근 방식만 6가지입니다. 각각이 "AI 기억이란 무엇인가"에 대해 다른 답을 갖고 있습니다.

## 접근 1: 절대 잊지 않는다

**elite-longterm-memory** 스킬의 철학입니다. 다운로드 9,500회를 기록한 이 스킬의 핵심은 5+1 레이어 아키텍처입니다.

- **Hot RAM** — SESSION-STATE.md. 지금 무슨 작업 중인지.
- **Warm** — LanceDB 벡터 검색. 최근 기억을 의미 기반으로 탐색.
- **Cold** — Git-Notes 지식 그래프. 개념 간 관계 추적.
- **Archive** — MEMORY.md. 핵심 기억 인덱스.
- **Cloud** — SuperMemory로 외부 백업.
- **Auto-Extract** — Mem0가 대화에서 자동으로 기억을 추출.

가장 극단적인 기능은 **WAL(Write-Ahead Log) 프로토콜**입니다. 응답하기 전에 반드시 기록합니다. 데이터베이스에서 "크래시가 나도 데이터를 잃지 않는다"는 원칙을 AI 기억에 그대로 적용한 것입니다.

이 접근의 철학 한 줄: *아무것도 버리지 마라. 기억이 많을수록 더 유능하다.*

## 접근 2: 잊는 것도 능력이다

**memory-hygiene** 스킬은 반대 방향을 봅니다. LanceDB에 쌓인 벡터 메모리를 주기적으로 감사하고, 정리하고, 최적화합니다.

왜 잊어야 할까요? 오래된 기억이 현재 판단을 오염시키기 때문입니다.

"Tailwind CSS를 쓰기로 했다"는 기억이 6개월 전 것이라면, 지금은 틀릴 수 있습니다. 하지만 AI는 이걸 검색해서 "우리는 Tailwind를 씁니다"라고 자신 있게 답합니다. 불필요한 기억이 잘못된 회상(hallucination)을 유발하는 겁니다.

이 스킬은 월 1회 자동으로 메모리를 청소하는 크론 잡을 포함합니다.

이 접근의 철학 한 줄: *적게 기억하되 정확하게. 잊는 것은 최적화다.*

## 접근 3: 인간 두뇌를 모방한다

**cognitive-memory**는 가장 야심찬 접근입니다. 인간 인지과학의 메모리 구조를 그대로 에이전트에 이식합니다.

4개 메모리 스토어:
- **Episodic** — 사건 기억. "3월 5일 서버가 다운됐다."
- **Semantic** — 지식 그래프. "Mason은 리스크를 즐긴다."
- **Procedural** — 학습된 워크플로우. "보고서 작성 순서."
- **Vault** — 절대 삭제되지 않는 기억. "API 키 위치."

자연어로 조작합니다. "remember", "forget about", "reflect on" 같은 명령어가 동작합니다.

가장 흥미로운 기능은 **수면 시간 반성(reflection)**입니다. 에이전트가 혼자 기억을 정리하고, "이 패턴이 반복되네. 다음엔 이렇게 하자"는 메타 인지까지 수행합니다.

감쇠 점수 λ=0.03, 약 23일 반감기. 에빙하우스 망각곡선과 거의 같은 모양입니다.

## 접근 4: 파일이 진실이다 (OpenClaw V3)

OpenClaw 내장 메모리 시스템의 철학입니다.

- MEMORY.md + memory/YYYY-MM-DD.md = 마크다운 파일이 원본
- 벡터 DB는 검색용 인덱스에 불과
- 하이브리드 검색: 의미 기반 70% + 키워드 30%
- 시간 감쇠: 30일 반감기, 에버그린 파일은 면제

대부분의 AI 메모리 시스템(Mem0, Zep)은 임베딩 벡터만 저장합니다. 블랙박스입니다. AI가 뭘 기억하는지 사람이 확인할 방법이 없습니다.

OpenClaw는 다릅니다. `cat MEMORY.md`를 실행하면 AI의 기억이 전부 보입니다. `git diff`로 기억 변화를 추적할 수 있고, `vi MEMORY.md`로 직접 수정할 수 있습니다.

이게 핵심 차별점입니다. [메모리 컴팩션과 기억 손실에 대해서는 이 글](/blog/ai-memory-compaction)에서 자세히 다뤘습니다.

## 접근 5: 3계층 통합

**persistent-memory**는 L1 Markdown + L2 ChromaDB 벡터 + L3 NetworkX 지식 그래프를 조합합니다. 마크다운 파일이 변경되면 벡터와 그래프가 자동 동기화됩니다.

OpenClaw V3와 철학이 비슷하지만, 지식 그래프가 추가된 점이 다릅니다. "Mason이 Xitadel 대표다" → "Xitadel은 온체인 채권 회사다" → 관계 추론이 가능해집니다.

## 접근 6: 프레임워크에 종속되지 않는다

**memsearch**는 Milvus/Zilliz 팀이 OpenClaw의 메모리 아키텍처를 추출해서 독립 라이브러리로 만든 것입니다. 어떤 에이전트 프레임워크에든 플러그인 형태로 붙일 수 있습니다.

원래 설계가 좋으면 분리해서도 쓸 수 있다는 증거입니다.

## 시간 감쇠: AI도 옛 기억을 잊는다

OpenClaw V3의 시간 감쇠 공식입니다.

```
decay = e^(-λ × days)
λ = ln(2) / halfLifeDays (기본값: 30일)
```

실제로 계산해보면:

| 경과 기간 | 기억 강도 |
|----------|----------|
| 7일 전 | 84% |
| 30일 전 | 50% |
| 90일 전 | 12.5% |
| 180일 전 | 1.6% |

단, MEMORY.md에 기록된 에버그린 정보는 감쇠에서 제외됩니다. "당신의 타임존은 KST"는 6개월이 지나도 유지됩니다. "Tailwind 쓰기로 했다"는 3개월이면 거의 사라집니다.

## 기억과 작업 상태는 다르다

하나 더 짚어야 할 구분이 있습니다.

**기억**은 "10년 경력"처럼 이력서에 쓰는 것입니다. **작업 상태**는 "지금 이 버그 3시간째 잡는 중, A, B 다 해봤고 C만 남았다"는 것입니다.

대부분의 메모리 시스템은 기억을 저장하지만, 작업 상태는 별도로 관리해야 합니다. `/savestate`와 `/resume` 같은 수동 패턴이 필요한 이유입니다.

데이터베이스 용어로 비유하면, 기억은 스냅샷이고 작업 상태는 WAL입니다.

## 어떤 철학을 고를까

정답은 없습니다. 하지만 판단 기준 세 가지는 줄 수 있습니다.

1. **투명성이 필요하면** → OpenClaw V3 (파일이 진실)
2. **정교한 메모리 설계가 목표면** → cognitive-memory (인간 두뇌 모방)
3. **가볍게 시작하고 싶으면** → persistent-memory (3계층 통합)

"아무것도 안 하는 것"만이 분명한 오답입니다. 45시간의 기억을 잃은 경험이 있는 사용자라면 동의할 겁니다.
